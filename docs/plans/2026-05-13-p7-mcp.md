# P7 MCP Server Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans (or superpowers:subagent-driven-development) to implement this plan task-by-task.

**Goal**：交付 MCP (Model Context Protocol) server，讓 AI agent 能透過標準 MCP 介面安全存取主要業務能力（查詢訂單、下單、查詢稽核、停權商家、處理爭議），且所有 MCP 操作寫入 audit_event 並標記 `request_id` 為 `mcp:` 前綴。

**Architecture**：MCP server 跑在同一 binary（`--role=api` 順帶啟用），複用 identity/order/vendor/payroll/compliance Service 的 interface — 不重寫業務規則。HTTP transport (`/mcp` endpoint, SSE-based) + stdio transport (`--role=mcp-stdio`) 兩種啟動方式。權限沿用 Bearer token + role 檢查。

**Tech Stack**：`github.com/mark3labs/mcp-go` (廣泛使用的 Go MCP SDK)。沿用其他依賴。

**Branch**：`feat/p7-mcp`（已切）

**Scope boundary**：
- P7 **做**：MCP server SSE + stdio、9 tools (5 read + 4 write)、auth via existing Bearer + role guards、audit with request_id="mcp:<tool>"、unit + integration tests、HTTP `/mcp` route + new `--role=mcp-stdio` entrypoint
- P7 **不**做：MCP resources (URI-based read like `vendor://{id}`)、prompts、scoped sub-tokens（用既有 admin token；後續可加 scope claims）、SSO 流程整合（agent 帶既有 token）

**Step 2**：`server.go`:

```go
package mcpserver

import (
    "context"

    "github.com/mark3labs/mcp-go/mcp"
    "github.com/mark3labs/mcp-go/server"

    "github.com/takalawang/corporate-catering-system/services/api/internal/identity"
    "github.com/takalawang/corporate-catering-system/services/api/internal/order"
    "github.com/takalawang/corporate-catering-system/services/api/internal/payroll"
    vendor "github.com/takalawang/corporate-catering-system/services/api/internal/vendors"
    "github.com/takalawang/corporate-catering-system/services/api/internal/compliance"
)

// Deps carries the underlying services the MCP layer reuses.
// MCP tools call these Services directly — same business rules as HTTP.
type Deps struct {
    Identity   *identity.Service
    Order      *order.Service
    Vendor     *vendor.Service
    Payroll    *payroll.Service
    Compliance *compliance.Service
    Users      identity.UserRepository
    Sessions   identity.SessionStore
}

func New(deps Deps) *server.MCPServer {
    s := server.NewMCPServer(
        "T-Bite MCP", "0.1.0",
        server.WithToolCapabilities(true),
    )
    registerOrderTools(s, deps)
    registerVendorTools(s, deps)
    registerPayrollTools(s, deps)
    registerAuditTools(s, deps)
    return s
}

// helper: extract authenticated user from MCP context.
// We attach the user upstream via context.WithValue when the HTTP transport
// wraps requests; stdio mode runs as a single authenticated identity (env var).
func userFromCtx(ctx context.Context) (*identity.User, bool) {
    u, ok := ctx.Value(userCtxKey{}).(*identity.User)
    return u, ok
}

type userCtxKey struct{}

// AttachUser wraps a context with the authenticated user for tool handlers.
func AttachUser(ctx context.Context, u *identity.User) context.Context {
    return context.WithValue(ctx, userCtxKey{}, u)
}
```

`registerOrderTools` / `registerVendorTools` / `registerPayrollTools` / `registerAuditTools` will be filled in subsequent tasks. For Task 1, define them as no-ops (or skip the call) and just have New return a server with no tools yet.

**Step 3**：HTTP mount

`services/api/internal/httpserver/server.go` — accept an optional `*server.MCPServer` and mount `/mcp`:

```go
func New(addr string, logger *slog.Logger, idAPI *idhttp.API, extra func(chi.Router), mcp *server.MCPServer, apiBuilders ...func(huma.API)) *Server {
    r := chi.NewRouter()
    // ... existing middleware + huma ...

    if mcp != nil {
        // Wrap MCP HTTP handler with auth middleware:
        // The MCP handler receives a request; we authenticate it via Bearer token
        // and attach the User to the request context before invoking the MCP handler.
        sseServer := server.NewSSEServer(mcp)
        r.Handle("/mcp/*", idAPI.AuthMiddleware(sseServer))
    }
    // ... rest
}
```

The `server.NewSSEServer(mcp)` returns an `http.Handler` that mounts MCP's SSE protocol. `idAPI.AuthMiddleware` (from P1) already injects the User into ctx — so MCP tool handlers can call `idhttp.UserFromContext(ctx)`.

But — MCP tools need `*identity.User` from `mcpserver.userFromCtx`. Adapter: after `idhttp.AuthMiddleware`, add a tiny middleware that reads `idhttp.UserFromContext(ctx)` and re-attaches via `mcpserver.AttachUser(ctx, u)`. Or simpler: have `userFromCtx` in `mcpserver` call `idhttp.UserFromContext` directly.

Choose **simpler**: `mcpserver.userFromCtx` delegates to `idhttp.UserFromContext`:

```go
func userFromCtx(ctx context.Context) (*identity.User, bool) {
    return idhttp.UserFromContext(ctx)
}
```

Drop the AttachUser indirection. Update Step 2 accordingly.

Sketch — `tools_order.go`:

```go
package mcpserver

import (
    "context"
    "encoding/json"
    "fmt"

    "github.com/mark3labs/mcp-go/mcp"
    "github.com/mark3labs/mcp-go/server"

    "github.com/takalawang/corporate-catering-system/services/api/internal/identity"
)

func registerOrderTools(s *server.MCPServer, deps Deps) {
    s.AddTool(
        mcp.NewTool("order.list_mine",
            mcp.WithDescription("List the authenticated employee's recent orders (last 30 days)"),
            // No input args needed
        ),
        func(ctx context.Context, req mcp.CallToolRequest) (*mcp.CallToolResult, error) {
            u, ok := userFromCtx(ctx)
            if !ok { return mcp.NewToolResultError("not authenticated"), nil }
            if u.Role != identity.RoleEmployee {
                return mcp.NewToolResultError("employee role required"), nil
            }
            orders, err := deps.Order.ListByUser(ctx, u.ID)
            if err != nil { return mcp.NewToolResultError(err.Error()), nil }
            // Write audit
            requestID := "mcp:order.list_mine"
            // Don't fail the tool on audit failure; log instead
            // For simplicity: omit audit write here (audit covered in Task 5)
            _ = requestID
            data, _ := json.Marshal(map[string]any{
                "count": len(orders),
                "orders": orders,
            })
            return mcp.NewToolResultText(string(data)), nil
        },
    )

    s.AddTool(
        mcp.NewTool("order.get",
            mcp.WithDescription("Get an order by ID. Owner can fetch their own; admin can fetch any."),
            mcp.WithString("order_id", mcp.Required(), mcp.Description("UUID of the order")),
        ),
        func(ctx context.Context, req mcp.CallToolRequest) (*mcp.CallToolResult, error) {
            u, ok := userFromCtx(ctx)
            if !ok { return mcp.NewToolResultError("not authenticated"), nil }
            orderID, err := req.RequireString("order_id")
            if err != nil { return mcp.NewToolResultError(err.Error()), nil }
            if u.Role == identity.RoleEmployee {
                o, err := deps.Order.Get(ctx, orderID, u.ID)
                if err != nil { return mcp.NewToolResultError(err.Error()), nil }
                data, _ := json.Marshal(o)
                return mcp.NewToolResultText(string(data)), nil
            }
            if u.Role == identity.RoleWelfareAdmin {
                // Admin can fetch any order; use a direct repo lookup via Service if available
                // For P7, expose this via order.Service — add a "AdminGet" method or just use existing Get with empty user check
                return mcp.NewToolResultError("admin order.get not implemented in P7 (use HTTP API)"), nil
            }
            return mcp.NewToolResultError(fmt.Sprintf("role %s cannot get orders", u.Role)), nil
        },
    )
}
```

## Task 5：MCP audit integration

Wire audit write into every tool handler. The audit signature is the same as HTTP: 
- `action = "mcp." + toolName`
- `target_kind / target_id` from tool inputs (e.g. vendor_id, order_id, batch_id)
- `request_id = "mcp:" + toolName`
- `actor_id` from authenticated user; `actor_role` matches user.Role

Refactor: add a helper `auditAfter(ctx, deps, action, targetKind, targetID, payload, user)` that runs the audit write in its own tx. Call from each tool handler post-success.

Tests: integration test that runs MCP tool against testcontainers Postgres + asserts `audit_event` row appears with `request_id` starting with `mcp:`.

```go
case config.RoleMCPStdio:
    // Auth via env: MCP_BEARER_TOKEN
    token := os.Getenv("MCP_BEARER_TOKEN")
    if token == "" { logger.Error("MCP_BEARER_TOKEN required for stdio mode"); os.Exit(2) }

    pool, _ := db.NewPool(ctx, cfg.DatabaseRW)
    defer pool.Close()
    rdb, _ := cache.NewClient(ctx, cfg.RedisURL)
    defer rdb.Close()

    // Build all services (same as api role)
    // ... [identical to api role] ...

    // Resolve the user from the env token
    sess, err := sessStore.Get(ctx, token)
    if err != nil { logger.Error("token invalid"); os.Exit(1) }
    user, err := userRepo.GetByID(ctx, sess.UserID)
    if err != nil { logger.Error("user lookup"); os.Exit(1) }

    // Run MCP server on stdio with user pre-attached to every tool call
    mcpSrv := mcpserver.New(deps)
    stdioServer := server.NewStdioServer(mcpSrv)
    // Wrap stdio context with the user
    if err := stdioServer.Listen(ctx, /* with user in ctx */); err != nil {
        logger.Error("stdio", "err", err); os.Exit(1)
    }
```

Note: mcp-go's stdio server may not directly support context middleware. Workaround: wrap the server to inject user into request ctx via a tool-call interceptor. If mcp-go API doesn't support that, simplest hack: stdio mode runs in a goroutine + a global atomic `currentUser` (acceptable since stdio is single-process single-user).
