# ADR 0002 — Helm Umbrella Chart as Canonical Packaging

- **Status**: Accepted — 2026-05-25
- **Source issue**: [Agentic-Build/corporate-catering-system#49](https://github.com/Agentic-Build/corporate-catering-system/issues/49)
- **Parent baseline**: [`00-baseline.md`](00-baseline.md)

## Context

The corporate catering platform is not a single application deployment.
It is a product-shaped self-hosted stack composed of application
services, ingress, transactional and read-replica databases, an event
bus with durable streams, object storage, an observability backend,
secret materials, schema migrations, and provisioning jobs that
materialize the runtime preconditions of the application services. A
production installation must coordinate the versions of each
constituent component, enforce the configuration surface that is
required for the installation to function correctly, and remain
repeatable across heterogeneous customer environments. These
requirements describe a packaging problem, not an application-code
problem, and they apply equally to development, staging, and production
installations.

A packaging baseline that addresses the product as a whole must
therefore offer three properties: versioned dependencies that can be
pinned and vendored, an explicit configuration contract that validates
required inputs before any resource is rendered, and repeatable
installation semantics that produce the same Kubernetes resources for
the same inputs across environments. The architectural pressure is to
turn the product release into a release boundary rather than a
collection of independently managed services that customers must
assemble.

## Decision

The canonical packaging format is a Helm umbrella chart.

The chart supports two operating modes. The first, *batteries-included
self-host mode*, is the canonical path and installs the selected
self-hosted dependencies declared in [`adr-0004`](adr-0004-self-hosted-ha-data-plane.md).
The second, *BYO mode*, supplies externally managed endpoints through
values while preserving the same application contracts. Application
code is identical in both modes.

## Rationale

Helm provides the stack with a release boundary that aligns with the
product distribution shape. Pinned dependency versions make
reproducibility a property of the chart rather than of the customer's
discipline. Values schemas allow the chart to refuse incomplete or
contradictory configuration before any resource is rendered, which
prevents an entire class of misconfiguration from reaching the
Kubernetes API. Upgrade history is recorded by Helm itself, which gives
operators a predictable rollback model. Runtime performance is
unaffected because Helm renders Kubernetes resources before deployment;
the choice of Helm is therefore about packaging discipline, dependency
management, and operational repeatability rather than about runtime
characteristics.

The principal alternative considered and rejected was Kustomize as the
sole packaging mechanism. Kustomize is excellent for site-specific
post-render patches but does not natively express versioned external
dependencies, does not provide a values-schema validation surface
comparable to `values.schema.json`, and does not produce a release-history
boundary equivalent to Helm releases. Adopting Kustomize as the sole
packaging mechanism would require building a parallel mechanism for
each of those concerns. The decision retains Kustomize as a permitted
post-render tool for site-specific patches but rejects it as the
carrier for the dependency graph or the product release contract. A
second alternative considered was raw Kubernetes manifests generated by
a custom templating tool; that alternative was rejected because it
would reproduce the responsibilities of Helm without the corresponding
ecosystem support.

## Design Implications

The chart introduces the following artifacts and discipline. The values
schema `values.schema.json` defines required configuration and rejects
invalid combinations, which prevents misconfiguration from reaching the
cluster. Self-hosted dependencies declared in [`adr-0004`](adr-0004-self-hosted-ha-data-plane.md)
are version-pinned in `Chart.yaml` and must be vendorable for offline
installation, satisfying the air-gap requirement of
[`adr-0006`](adr-0006-secrets-and-air-gap.md). BYO mode must not introduce
application-code branches for managed cloud services; the application
consumes the same connection URLs, the same readiness contracts, and
the same telemetry contracts in both modes, with the only difference
being the source of the underlying endpoints. Migration and
provisioning jobs (governed by [`arch-0007`](arch-0007-cloud-native-readiness-and-autoscaling.md))
are represented as chart-managed Kubernetes Jobs so that installation
and upgrade behavior is observable and repeatable.

The chart is rendered and installed through the Makefile targets
`chart-render` and `chart-install`, which provide a single command
surface for both local and production installation flows.

## Acceptance Criteria

- One chart family can install the app stack and canonical self-host dependencies.
- The same chart supports local, staging, and production values.
- Third-party chart versions are pinned.
- Configuration required for production is validated by schema.
- Kustomize use is limited to site-specific patching or post-render customization.

## Compliance Evidence

| Acceptance criterion | Compliance |
| --- | --- |
| One chart family can install the app stack and canonical self-host dependencies. | `chart/tbite-platform/` contains the umbrella chart with application templates and dependency declarations for CloudNativePG, NATS, MinIO Operator, Valkey, Traefik, cert-manager, and the Victoria stack. |
| The same chart supports local, staging, and production values. | `chart/tbite-platform/values.yaml`, `chart/tbite-platform/values-dev.yaml`, and `chart/tbite-platform/values-prod.yaml` share schema and structure. |
| Third-party chart versions are pinned. | `chart/tbite-platform/Chart.yaml` declares each dependency with an explicit version specifier. |
| Configuration required for production is validated by schema. | `chart/tbite-platform/values.schema.json` declares required production fields and rejects invalid combinations at `helm template`/`helm install` time. |
| Kustomize use is limited to site-specific patching or post-render customization. | The chart is the sole release artifact; no Kustomize overlay is part of the production contract. Site-specific patches may be applied via Helm post-render hooks. |

## Scope Boundary

This ADR does not decide the ArgoCD Application, ApplicationSet, or
app-of-apps layout. It only states that the produced chart must be
compatible with GitOps operation. The selection of self-hosted
dependencies is delegated to [`adr-0004`](adr-0004-self-hosted-ha-data-plane.md).
The selection of the runtime substrate is governed by
[`adr-0001`](adr-0001-kubernetes-only-runtime.md).
